# 渲染管线重构 + 玻璃板修复 + 文档同步

## 第一阶段：玻璃板"只渲染中心"问题修复

### Task 1.1 诊断结论
根因是 **multipart 烘焙时的双重重置**：
- `VanillaModelManager.ModelBaking.bakeBlockstateForBlock()` 对 multipart 模型，将各部分的 `apply.y` 旋转记录为全局 `rotation` 变量，存入 `rotMap`
- 但 `bakeModel()` 不实际应用旋转（`rotationY` 参数仅用于缓存key）
- 结果是 render 时所有 quad 被整体旋转，侧板移到错误方向，仅中心柱 stay

**示例**：glass_pane meta=1（north=true）：
- post(无旋转) + side(无旋转) + noside_alt(无旋转) + noside(y=180记录全局) + noside_alt(y=180覆盖全局)
- rotMap[1]=180 → render时全部旋转180度
- side面板(7-9 x, 0-7 z) → 旋转到南侧(7-9 x, 9-16 z)，侧板消失

### Task 1.2 修改 bakeBlockstateForBlock 的 multipart 分支
文件：`VanillaModelManager.java`
- 在 multipart 烘焙中，当 `bakeModel()` 返回 quads 后，对 `mpc.apply.y != 0` 的 part 做实际旋转
- 始终设置 `rotMap.put(meta, 0)`
- 全局旋转不应用于 multipart 模型

在 `BlockJsonModelBake` 中添加静态方法 `applyYRotation(List<BakedQuad>, int degY)`，对 quad 顶点做 Y 轴旋转。

### Task 1.3 验证
- 创建测试场景：放置无色玻璃板，连接不同方向的方块
- 确认面板正确延伸，中心柱和侧板正常渲染
- 检查染色玻璃板同样正常工作

## 第二阶段：UniformRenderPipeline 重构——职责下沉至Extension

### 现状分析
`UniformRenderPipeline.java` 当前承担职责（573行）：
1. computeVertexAO() 逐顶点AO计算（复杂算法）
2. shadeByNormal() 方向阴影计算
3. getFaceBrightness() 面亮度查询
4. applyDisplayTransform() GL显示变换
5. GL_LIGHTING 开关管理（front/side检测）
6. 扩展链调度
7. Tessellator 提交（含AO/非AO分支）

重构目标：1-5全部下沉到Extension，管线只做 调度+提交。

### Task 2.1 创建 AoComputeExtension
新建文件：`extension/AoComputeExtension.java`
- 实现 IModelRenderExtension，**强制作为扩展链第一个扩展**
- 在 `apply()` 中对 BLOCK_WORLD 阶段执行逐顶点 AO 计算
- 从 RenderContext 读取 world/x/y/z/block/quad
- 将计算结果写入 ctx.aoBrightness[4] 和 ctx.aoColorMul[4]
- 整个 `computeVertexAO()` 算法逻辑 + 辅助方法移入此类

### Task 2.2 创建 DisplayTransformExtension
新建文件：`extension/DisplayTransformExtension.java`
- 实现 IModelRenderExtension
- 在 apply() 中根据 phase 映射到 display key（gui/firstperson/thirdperson）
- 检查 RenderContext.quad 是否有 display data
- 直接应用 GL11.glTranslatef/glRotatef/glScalef
- 注意：display transform 需要在 Tessellator 之前推入 GL 矩阵

**设计约束**：DisplayTransformExtension 需要 access 到模型的 display transform 数据。
- 当前 display 通过 `ItemModelWrapper` 传入 `UniformRenderPipeline.renderItemQuads(part, stack, phase, display)`
- 重构后：display 数据应存储在 BakedQuad 或通过全局注册中心

**方案**：在 `ModelJson` 新增 `Map<String, DisplayTransform> display` 字段（已有），烘焙时将其挂载到所有 BakedQuad 上（类似 guiLight 传播），新增 `BakedQuad.modelDisplay` 字段。

### Task 2.3 将 GL_LIGHTING 管理移至 GuiLightExtension
修改：`GuiLightExtension.java`
- 当前只处理 shade（front→1.0）
- 新增 GL_LIGHTING 状态管理能力
- 由于 apply() 在每个 quad 上调用，直接开关 GL_LIGHTING 效率低
- 方案：新增 `IModelRenderExtension.beforePart()` 生命周期 + `afterPart()` 清理
- 在 `beforePart` 中检测 guiLight 并执行 GL11.glDisable/Enable
- 在 `afterPart` 中恢复状态

但此改动涉及接口变更，影响面大。替代方案：
- GuiLightExtension 保持只处理 shade
- 在 UniformRenderPipeline 中添加对 GuiLightExtension 的特殊调度逻辑
- 或者：在 ModelRenderRegistry 中添加 `beforePart()` 回调机制

**最终方案**：采用更轻量的方式——在 UniformRenderPipeline 中将 GL_LIGHTING 逻辑委托给 GuiLightExtension 通过静态方法调用（不改变扩展接口）。

### Task 2.4 精简 UniformRenderPipeline
修改：`UniformRenderPipeline.java`
- 删除 `computeVertexAO()` 及其所有辅助方法
- 删除 `applyDisplayTransform()`
- 删除 `getFaceBrightness()` 
- 删除 `shadeByNormal()`
- 保留：创建 RenderContext → 调用 ModelRenderRegistry.apply()（含新扩展）→ Tessellator 提交
- 保留：四方块旋转（这是渲染时坐标变换，无法下沉）
- 保留：Tessellator 提交的分支逻辑（有 AO vs 无 AO）

精简后的管线约 150 行（原 573 行）。

### Task 2.5 更新 ModelRenderRegistry.ensureDefaults() 注册顺序
修改：`ModelRenderRegistry.java`
- 确保 AoComputeExtension 为链头第一扩展
- DisplayTransformExtension 在合适位置
- GuiLightExtension 保持
- 调整 import
